## 2. Project Understanding

### 2.1 これは何か
- 3D野球ミニゲーム。2モード構成:
  1. **工房モード**: 回転する丸太を鉋カーソルで削ってバットを成形（`radii[]` 配列が形状の真実）。質量・重心・慣性モーメント・芯(COP)・しなりを実計算し、レーダーチャートに表示。
  2. **打席モード**: 右打者。10球チャレンジ。2秒のワインドアップゲージ→投球→クリック/スペースでスイング。バット面の法線で反射計算した打球を追跡カメラで追い、飛距離をリアルタイム表示。終了後に総飛距離を localStorage ランキングに登録。
- 配布形態: **HTML 1ファイル + CDNのThree.js r128**。ビルド工程なし。`python3 -m http.server 8765` で開発（`.claude/launch.json` の `bat-game` 設定）。

### 2.4 データフロー

## 3. Behaviors To Preserve（絶対に壊してはいけない挙動）

1. **工房**: 左ボタン長押しで削れる／Shiftで戻る／ホイールで刃幅(2-10)／丸太が回転し続ける／計測パネルとレーダーチャートが削るたび更新される。
2. **打席フロー**: 工房→「完成！打席に立つ」→約1秒後に1球目のワインドアップ開始→青いゲージが2.0秒で縮み、消えた瞬間に発射→10球で終了画面。
3. **球速カーブ**: `62 + 球数*4.8 + rand*4` km/h（1球目 約67-71、10球目 約110-114）。**変更禁止**。
4. **スイング**: クリック/スペースで即始動（遅延を入れない）。スイングアシスト（668-671行）の吸い付き量・上限を変えない。
5. **当たり判定の甘さ**: 捕捉半径 `radiusAt(x) + BALL_R + 0.048`（712行）。±40ms/±10cmの誤差でほぼ全球コンタクトする現在の難易度を維持。
6. **打球計算の数値**: ゲーム性補正の lerp(0.8q)（731-736行）、球速ボーナス（737行）、指数倍率 `exp(max(0,E-0.82)*3.2)` 上限12・出球320m/s上限・`dragK=0.0042/mult`（738-757行）。**1つでも数値を変えると手触りが変わる。変更禁止。**
7. **判定しきい値**: ヒット≥25m、HR≥98m、場外≥140m、宇宙≥1000m、フェア=|spray|≤46°かつ lz<0（779-790行）。
8. **追跡カメラ**: ヒット瞬間からボール追尾＋飛距離リアルタイム更新、着地後に次球で打席視点へ復帰。長押しドラッグの視点オフセット。
9. **UI**: 操作説明は折りたたみ（❓ボタン）。終了画面で総飛距離表示→名前入力→ランキング登録（ボタン無効化で二重登録防止）。
10. **localStorage スキーマ**: キー `batcraft_rank`、要素 `{name, total, date}`、保存上限30件・表示10件。**既存保存データが読めなくなる変更は禁止。**
11. **効果音**: 削り/スイング/打撃(qで音程変化)/HR/ジャストミート/投球/着地/ファウル/UI の鳴り分け。
12. **モード切替の安全性**: 「工房に戻る」で投球・スイング・飛球・ワインドアップが完全停止する（953行）。
13. **デバッグフック `window.__bat` と `window.__lastHit`**: 検証ハーネスが依存している。削除・改名しない（§5参照）。

### D2. setTimeout チェーンとモード状態の併存（最危険箇所）
- **根拠**: 859行（空振り後の次球）、795行（着地後）、802行（nextPitch→startPitch）、832行（再戦）、993行（toGame→初球）、997行（retry）。`toCraft()`（951-953行）は状態変数を null にするが**ペンディング中の setTimeout は解除しない**。
- **なぜ負債か**: モード切替後にタイマーが発火する。現在は `startPitch` 内のガード（677行 `mode !== 'game' || windup || pitch`）で実害を防いでいるが、ガード頼みで暗黙的。過去に「工房に戻っても投球が続く」バグが実際に起きた箇所。
- **影響範囲**: 進行フロー全体。**リスク**: 中（タイミング挙動を変えると間合いが変わる）。
- **改善案**: ゲーム進行用タイマーを一元管理する小さなユーティリティ（`gameTimeout(fn, ms)` が ID を配列に記録し、`clearGameTimers()` を `toCraft`/`resetScore` から呼ぶ）。間合いの ms 値は一切変えない。
- **検証**: スモーク通過＋手動で「ワインドアップ中に工房へ戻る」「着地直後に戻る」「終了画面で戻る」の3操作でその後投球が発生しないこと。
- **判断**: **実装してよい**（ms値の変更は禁止）。

### D3. ミート幾何の定数が4箇所に分散
- **根拠**: ① スイングアシスト `0.94*BS.cop`・`0.40*BS.swingT`（668-670行）、② ミートリング `batPointAt(0.38, a, BS.cop)`（913行）、③ 軌道の水平点 `Math.pow(0.70, 3.5)`（650行）、④ 実接触は raw s≈0.3-0.4 で起こるという暗黙の事実（705行のガードと§2.4）。
- **なぜ負債か**: 「接触はスイング進行 s≈0.38 付近・バット方向 cos(yaw)≈0.94 で起こる」という同じ物理的事実を4箇所が別々の即値で持っており、1つだけ調整すると照準・リング表示・アシストがズレる（実際に開発中ズレによる空振り多発が起きた）。
- **影響範囲**: 照準・アシスト・リング表示。**リスク**: 中（値を1つでも変えると難易度が変わる）。
- **改善案**: `const MEET = {sE:0.38, dirX:0.94, leadT:0.40}` のような定数オブジェクトに集約し、4箇所から参照。**数値自体は現状のまま**。リング位置とアシストが同一のヘルパー（例 `meetPointFor(a)`）を共有する形が望ましい。
- **検証**: スモークのコンタクト率が baseline と同等（9-10/10）。リングの表示位置が目視で変わっていない。
- **判断**: **実装してよい**（数値変更は禁止）。

### D4. 工房⇄打席切替での GPU リソースリーク
- **根拠**: `buildField()`（399行）と `buildCraft()`（301行）は冒頭で `group.clear()` するが、Three.js r128 の `clear()` は子の geometry/material/texture を **dispose しない**。`buildField` は毎回 観客1200点・星900点・ビル26個・canvasテクスチャ数枚を新規生成する。`rebuildBatMesh`（341行）だけは正しく dispose している。
- **なぜ負債か**: 工房⇄打席を往復するたびに GPU メモリが増える。長時間プレイ（公開を想定）で劣化。
- **影響範囲**: モード切替。**リスク**: 低-中（dispose 漏れ・二重 dispose に注意）。
- **改善案**: (a) `disposeGroup(group)`（traverse して geometry/material/map を dispose）を clear 前に呼ぶ。または (b) 静的な球場を初回のみ構築し再利用（こちらは挙動差が出やすいので (a) を推奨）。`window.__bat` に `info: () => renderer.info.memory` を追加して計測可能にする。
- **検証**: 5往復後の `renderer.info.memory.geometries` が往復前後でほぼ一定（baseline では単調増加するはず。まず baseline で増加を確認してから直す）。
- **判断**: **実装してよい**（(a) の方式で。(b) は提案のみ）。

### D5. `tryContact()` の責務過多
- **根拠**: 704-767行。衝突判定・反発物理・ゲーム性補正・球速ボーナス・指数倍率・ジャストミート演出・HUD更新・効果音・fly生成・デバッグ記録を1関数で実施。
- **なぜ負債か**: バランス調整のたびにこの関数を読む必要があり、どの行が「物理」でどの行が「演出」か区別しにくい。今後の調整要求（このプロジェクトは頻繁にある）のたびに事故リスク。
- **影響範囲**: 打撃の全て。**リスク**: 中。順序依存（補正→ボーナス→倍率→キャップの適用順で結果が変わる）を**絶対に**変えないこと。
- **改善案**: 同一ファイル内で純粋関数に分離: `detectContact(swing, ball) → {x, n, dist} | null`、`exitVelocity(contact, vBall, vBat, BS, kmh) → {vOut, q, superHit, mult}`、呼び出し側で HUD/音/fly を処理。計算順序と数値は完全維持。
- **検証**: スモーク10球の `__lastHit` の `q`/`vOut` 分布が baseline と同等。可能なら分離前に固定入力での出力スナップショットを console で取り、分離後に一致確認（`pitchPos`/乱数を含まない単体呼び出しで比較できる形にする）。
- **判断**: **実装してよい**（Phase 4。順序・数値の同一性をスナップショットで証明すること）。

### D6. `buildField()` 235行の一枚岩
- **根拠**: 398-633行。地形・装飾・打者・UI的3Dオブジェクト（リング/ゲージ）・ボールが混在。
- **なぜ負債か**: 「観客を増やす」「ビルを変える」のような装飾変更が、ゲームに必須なオブジェクト（ball/aimRing/pitchGauge）の生成と同居していて事故りやすい。
- **影響範囲**: 球場描画。**リスク**: 低（純粋な抽出なら）。
- **改善案**: `buildField` 内をセクションごとの関数（`addGround()`, `addStands()`, `addSkyline()`, `addGameObjects()` 等）に**同一ファイル内で**抽出。生成順序は維持。
- **検証**: スクリーンショット比較（目視）＋スモーク通過。
- **判断**: **実装してよい**（Phase 5）。

### D7. フェア/ファウル角度の不一致（仕様不明）
- **根拠**: フェア判定 `|spray| <= 46`（779行）に対し、ファウルライン描画は±45°（433-437行）、ファウル陰影も±45°相当（414行 `CircleGeometry(280,64, 0.75π, 1.5π)`）。
- **なぜ負債か**: 見た目のラインと判定が1°ズレている。ライン際の打球で「見た目フェアなのにファウル」（逆も）が起こりうる。ただし 46° が「ライン上はフェア」の意図的なマージンの可能性もある。
- **影響範囲**: 判定の境界のみ。**リスク**: 低だがゲームルールに直結。
- **改善案**: 定数 `FAIR_DEG` に一本化。**ただしどちらの値（45/46）に揃えるかは仕様判断**。
- **検証**: 境界角の打球で見た目と判定が一致。
